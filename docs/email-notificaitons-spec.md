# Email Notification Specification

## Overview

The application should send email notifications for recurring obligations that are upcoming, due, or overdue.

For Version 1, email is the only notification channel.

Notifications should be generated by a scheduled command-line PHP script that runs once per day. The notification system should be designed so that additional notification channels (push notifications, SMS, in-app notifications, etc.) can be added in the future without changing the reminder logic.

---

# Daily Notification Runner

Create a command-line PHP script, for example:

```
bin/send_daily_notifications.php
```

The script should be executed once per day via cron.

Example:

```cron
0 7 * * * /usr/bin/php /path/to/familyoffice/bin/send_daily_notifications.php
```

The script should:

1. Load the application configuration.
2. Connect to the database.
3. Find all recurring obligations that require notification.
4. Determine which users should receive each notification.
5. Group notifications by recipient.
6. Send a single daily digest email to each recipient.
7. Record every notification that was successfully sent.
8. Log execution statistics and any errors.

The script should be **idempotent**. It must be safe to run multiple times without sending duplicate notifications.

---

# Notification Types

The notification engine should support the following types:

## Upcoming Reminder

Sent before an obligation is due.

Examples:

- Property taxes due in 14 days
- Passport renewal due in 60 days
- HVAC filter replacement due in 7 days

---

## Due Today

Sent when an obligation is due today.

---

## Overdue Reminder

Sent when an obligation is overdue.

Examples:

- Umbrella insurance renewal is 5 days overdue
- Dental cleaning is 30 days overdue

---

# Notification Settings

Each recurring obligation should include notification settings.

Suggested fields:

- Reminder lead time (days before due date)
- Responsible person

---

# Recipient Rules

Normally, notifications are sent to the responsible person.

If no responsible person has been assigned, notifications should be sent to the family administrator(s).

If an obligation applies to a child, notifications should still be sent to the responsible adult.

---

# Email Format

Each recipient should receive a single daily digest email rather than one email per obligation.

Example subject:

```
Family Office: 4 reminders for today
```

Example body:

```
You have 4 Family Office reminders.

Due Today
- Pay property taxes (link)
- Replace HVAC filter (link)

Upcoming
- Dental cleaning for Charlie — due in 7 days (link)
- Umbrella insurance renewal — due in 14 days (link)

Overdue
- Review estate planning documents — 12 days overdue (link)

View all reminders:
https://familyoffice.brianrosenthal.org
```

---

# Preventing Duplicate Notifications

Every notification that is sent should be recorded in a notification log.

Before sending a notification, the system should verify that the same notification has not already been sent for:

- Obligation
- Recipient
- Notification type
- Notification date

This allows the notification runner to be safely executed multiple times.

Suggested notification log fields:

- id
- family_id
- obligation_id
- recipient_user_id
- notification_type
- notification_date
- sent_at
- email_address
- delivery_status
- error_message

---

# Logging

The notification runner should log:

- Script start time
- Script completion time
- Number of obligations evaluated
- Number of notifications generated
- Number of emails sent
- Delivery failures
- Unexpected errors
